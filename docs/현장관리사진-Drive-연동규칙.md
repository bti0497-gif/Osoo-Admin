# 현장관리 사진 Google Drive 연동 규칙

> 대상: Osoo Admin App 개발자 및 중앙관리자  
> 기준일: 2026-06-23  
> 현장 앱 기준 커밋: `4fc1205`

## 1. 목적

현장관리자 앱은 업무 중 생성되는 사진을 로컬 PC에 먼저 저장하고 Google Drive에 업로드합니다.

중앙관리자 앱은 이 사진을 새로 올리거나 수정하는 앱이 아닙니다. Drive에서 현장별·기간별 사진을 조회하고 내려받아 공사 월정산 자료와 관련 보고서를 작성하는 용도로 사용합니다.

성적서는 이 규칙의 관리사진에 포함하지 않습니다. 성적서는 중앙관리자 앱에서 별도로 업로드하고 기존 성적서 메뉴와 폴더 구조로 관리합니다.

## 2. 역할 구분

| 구분 | 현장관리자 앱 | 중앙관리자 앱 |
|---|---|---|
| 수질분석 측정사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 슬러지 반출사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 청소필증 사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 약품 입고사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 키트 입고사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 거래명세서 사진 | 업로드·로컬 관리 | 조회·다운로드 |
| 수질 성적서 | 업로드하지 않음 | 업로드·조회·다운로드·관리 |

## 3. Drive 루트

현장관리 사진은 환경변수 `GOOGLE_DRIVE_FOLDER_ID`가 가리키는 공통 루트 아래에 저장됩니다.

```text
{GOOGLE_DRIVE_FOLDER_ID}/
└─ 관리사진/
   └─ YYYY/
      └─ MM/
         ├─ 2026-06-23_충청현장_슬러지반출1.jpg
         ├─ 2026-06-23_충청현장_청소필증.jpg
         ├─ 2026-06-23_충청현장_포도당.jpg
         └─ 2026-06-23_충청현장_암모니아성질소(NH3-N).jpg
```

날짜별 폴더나 현장별 폴더를 추가로 만들지 않습니다. 모든 현장의 해당 월 사진을 같은 월 폴더에 모으고 파일명으로 구분합니다.

## 4. 파일명 규칙

기본 규칙은 다음과 같습니다.

```text
YYYY-MM-DD_현장명_업무항목[순번].확장자
```

예:

```text
2026-06-23_충청현장_슬러지반출1.jpg
2026-06-23_충청현장_슬러지반출2.jpg
2026-06-23_충청현장_청소필증.jpg
2026-06-23_충청현장_포도당.jpg
2026-06-23_충청현장_거래명세서.jpg
2026-06-23_충청현장_암모니아성질소(NH3-N).jpg
2026-06-23_충청현장_수질분석_1차_암모니아성 질소.jpg
```

구성 요소:

- 날짜: `YYYY-MM-DD`
- 현장명: 현장 앱의 `app_settings.site_name`
- 업무항목: 사진 종류 또는 약품·키트·분석항목 이름
- 순번: 같은 날짜·현장·항목에 사진이 여러 장일 때만 사용
- 확장자: 수질 원본은 JPG 외 형식일 수 있으며, 약품·키트·슬러지 사진은 앱에서 JPG로 정규화

Windows와 Drive에서 사용할 수 없는 다음 문자는 `_`로 치환됩니다.

```text
\ / : * ? " < > |
```

## 5. 사진 종류별 업무항목

### 슬러지 반출

```text
YYYY-MM-DD_현장명_슬러지반출1.jpg
YYYY-MM-DD_현장명_슬러지반출2.jpg
```

같은 날짜에 반출사진을 추가하면 순번이 증가합니다.

### 청소필증

```text
YYYY-MM-DD_현장명_청소필증.jpg
```

같은 날짜·현장의 청소필증을 다시 저장하면 새 파일을 만들지 않고 동일한 Drive 파일을 업데이트합니다.

### 약품 입고

업무항목에는 설정된 약품명을 그대로 사용합니다.

```text
YYYY-MM-DD_현장명_포도당.jpg
YYYY-MM-DD_현장명_중탄산나트륨.jpg
YYYY-MM-DD_현장명_팩(PAC).jpg
YYYY-MM-DD_현장명_{추가약품명}.jpg
```

### 키트 입고

업무항목에는 설정된 키트명을 그대로 사용합니다.

```text
YYYY-MM-DD_현장명_암모니아성질소(NH3-N).jpg
YYYY-MM-DD_현장명_질산성질소(NO3-N).jpg
YYYY-MM-DD_현장명_인산염인(PO4-P).jpg
YYYY-MM-DD_현장명_알칼리도(ALK).jpg
```

### 거래명세서

```text
YYYY-MM-DD_현장명_거래명세서.jpg
```

### QnTECH 수질분석 측정사진

QnTECH 사진은 분석 출처 또는 차수와 분석항목을 함께 넣습니다.

```text
YYYY-MM-DD_현장명_수질분석_{출처·차수}_{분석항목}.확장자
```

예:

```text
2026-06-23_충청현장_수질분석_1차_암모니아성 질소.jpg
2026-06-23_충청현장_수질분석_MBR_질산성 질소.jpg
2026-06-23_충청현장_수질분석_알칼리도.jpg
```

출처·차수가 없는 경우 해당 부분은 생략될 수 있습니다.

## 6. 성적서 예외

성적서는 현장관리자 앱이 업로드하지 않습니다.

중앙관리자 앱의 기존 성적서 업로드·조회 로직과 다음 Drive 구조를 유지합니다.

```text
{CERTIFICATE_DRIVE_FOLDER_ID}/
└─ 성적서/
   └─ YYYY/
      └─ MM/
         ├─ 성적서_YYYYMMDD_현장명.jpg
         ├─ mlss_YYYYMMDD_현장명.jpg
         └─ 기타 성적서 파일
```

`관리사진`과 `성적서`의 검색·업로드·삭제 로직을 서로 섞지 않습니다.

## 7. 중앙관리자 앱 조회 권장 방식

월정산 화면에서는 다음 순서로 조회하는 것이 안전합니다.

1. 사용자가 연도와 월을 선택
2. `관리사진/YYYY/MM` 폴더 ID 조회
3. 해당 월 폴더의 파일을 한 번에 목록 조회
4. 파일명을 `_` 기준으로 최대 3개 영역으로 해석
5. 날짜·현장명·업무항목 필터 적용
6. 선택 파일을 원본 또는 ZIP으로 다운로드

파일명 해석 규칙:

```text
첫 번째 `_` 앞       → 날짜
첫 번째와 두 번째 `_` 사이 → 현장명
두 번째 `_` 뒤 전체   → 업무항목과 확장자
```

수질분석 파일은 업무항목 안에 `_`가 추가로 들어갈 수 있으므로 단순히 모든 `_`를 분할하면 안 됩니다.

권장 정규식:

```regex
^(\d{4}-\d{2}-\d{2})_([^_]+)_(.+)\.([^.]+)$
```

캡처 그룹:

1. 날짜
2. 현장명
3. 업무항목과 선택적 순번
4. 확장자

## 8. 월정산용 분류 권장값

중앙관리자 앱은 파일명의 업무항목을 다음처럼 분류할 수 있습니다.

| 판정 기준 | 분류 |
|---|---|
| `슬러지반출`로 시작 | 슬러지 반출사진 |
| `청소필증`과 일치 | 청소필증 |
| `거래명세서`와 일치 | 거래명세서 |
| `수질분석_`으로 시작 | QnTECH 수질분석 사진 |
| 키트 설정명과 일치 | 키트 입고사진 |
| 약품 설정명과 일치 | 약품 입고사진 |
| 그 외 | 기타 관리사진 |

약품명과 키트명은 앞으로 설정에서 변경되거나 추가될 수 있으므로 고정 목록만으로 제한하지 않는 것이 좋습니다.

## 9. BigQuery와 사진의 관계

현장 운영 수치는 BigQuery에서 조회하고 사진 원본은 Drive에서 조회합니다.

- QnTECH 수질값: `daily_log_system.qntech_water_quality`
- 약품 입고·사용·재고: `daily_log_system.medicine_logs`
- 키트 입고·사용·재고: `daily_log_system.kit_logs`
- 슬러지 반출량: `daily_log_system.flow_readings`
- 현장 업무사진: Drive `관리사진/YYYY/MM`

현장 업무사진은 모든 유형에 대해 Drive 파일 ID가 BigQuery에 저장된다고 가정하면 안 됩니다.

중앙관리자 앱은 다음 키로 수치와 사진을 연결해야 합니다.

```text
날짜 + 현장명 + 업무항목
```

수질분석 사진은 다음 키를 추가로 사용할 수 있습니다.

```text
날짜 + 현장명 + 분석항목 + 출처·차수
```

## 10. 중복과 업데이트 규칙

Drive 업로드 서비스는 같은 월 폴더에 동일한 파일명이 있으면 새 파일을 만들지 않고 기존 파일의 내용을 업데이트합니다.

따라서 중앙관리자 앱은 파일 ID가 바뀌지 않은 상태에서 파일 내용이나 수정 시각이 변경될 수 있음을 고려해야 합니다.

목록 캐시는 다음 경우 새로고침합니다.

- 사용자가 새로고침 버튼을 누른 경우
- 월정산 화면을 다시 연 경우
- 앱 시작 후 일정 시간이 지난 경우

## 11. 로컬 우선과 Drive 실패

현장 앱은 사진을 로컬에 먼저 저장합니다. Drive 업로드가 실패해도 현장 업무 데이터와 로컬 사진은 유지됩니다.

따라서 어떤 사진이 당장 Drive에서 보이지 않는다고 해서 현장 입력 자체가 실패했다고 판단하면 안 됩니다.

중앙관리자 앱에서는 다음 상태 구분을 권장합니다.

- Drive에 파일 있음: 조회·다운로드 가능
- BigQuery 수치는 있으나 Drive 사진 없음: `사진 미동기화 가능성` 표시
- Drive 사진은 있으나 대응 수치 없음: `사진만 존재` 표시

향후 현장 앱에 Drive 재전송 큐가 추가되면 중앙관리자 앱은 기존 경로와 파일명 규칙을 그대로 사용할 수 있습니다.

## 12. 기존 레거시 폴더

2026-06-23 이전 시험 데이터는 다음과 같은 옛 구조에 남아 있을 수 있습니다.

```text
{루트}/{현장명}/수질분석/YYYY/MM/YYYY-MM-DD/
{루트}/{현장명}/약품입고/YYYY/MM/YYYY-MM-DD/
{루트}/{현장명}/슬러지/YYYY/MM/YYYY-MM-DD/
```

신규 중앙관리자 기능은 `관리사진/YYYY/MM`을 기본 원천으로 사용합니다.

필요하면 별도의 레거시 검색 버튼이나 일회성 마이그레이션 도구를 제공할 수 있지만, 신규 사진을 옛 구조에 저장해서는 안 됩니다.
